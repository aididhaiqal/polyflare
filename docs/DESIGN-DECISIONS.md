## Q1 â€” Center of gravity â†’ **B: Dual-upstream, provider-neutral core**
The core is protocol-neutral from day one. A **translator registry** (`Format` enum + `(from,to) â†’ Translator` map, borrowed from CLIProxyAPI) is the spine; Codex and Anthropic are both **first-class backends** in the MVP, and any supported client protocol can route to either pool. codex-lb's Codex depth (fingerprint, quota-routing, continuity) becomes the Codex *backend implementation*, not the whole system.
- Implication: bigger MVP, but adding a provider is additive (register translators + an executor), never a rewrite.
- Rejected: (A) Codex-first (would need refactor to generalize later), (C) translator-first (defers the depth that's the actual value).

## Q2 â€” Deployment/coordination â†’ **A+: single-binary with cluster-shaped seams**
One tokio process, one embedded store, all state in-process. **Deletes codex-lb's entire multi-replica stack** (the ~8.4k-LOC bridge owner-forwarding subsystem, cap-partitioning, leader election, cache-invalidation poller, shared-Fernet key, advertise URLs) â€” all of which existed only because upstream targets team/k8s deployments the user doesn't run.
- Session-ownership + admission sit behind a `Coordinator` **trait** (in-process impl for v1) so a future distributed adapter is additive, not a rewrite. Revisit only if HA/multi-machine becomes a real need.
- "Multiple PCs" = multiple *clients* â†’ still ONE LB (network-reachable via LAN/Tailscale); a single multi-threaded tokio process (no GIL) handles heavy multi-client concurrency far better than Python codex-lb would. Cluster is only for HA of the LB box itself.
- Same posture as better-ccflare / CLIProxyAPI (single local process); codex-lb was the outlier.

## Q3 â€” MVP line â†’ **(i) Lean core, L0â€“L4**
First shippable = working dual-pool gateway: neutral core + translator registry (L0), Codex + Anthropic executors incl. the continuity state-machine + watchdog wedge-fix (L1), selector/pool/policy (L2), OpenAI-Responses + Anthropic-Messages ingress + Anthropicâ†’Codex translation (L3), thin observability + API-key auth + account OAuth/import (L4). **No dashboard in MVP** â€” it's the clean additive L5 (reads the same REST API); codex-lb remains the operator UI until then.

## C1 â€” Continuity engine (the wedge fix) â†’ **explicit per-conversation state machine + watchdog**
Model each conversation as an explicit state machine (`Fresh â†’ Anchored â†’ Reattaching â†’ {Anchored | Recover}`) owned in `polyflare-store` as `{anchor_response_id, owning_account, last_good_turn, reasoning_cache_ref, state}`, behind a `Continuity` trait (`prepare(req,state)â†’PreparedReq` / `observe(resp)â†’Transition`). Three rules: **(R1)** never trim a `store:false` full-resend to an anchor the upstream isn't guaranteed to hold; trim only when the client kept `store:true` history. **(R2)** every `Reattaching` gets an async watchdog â€” no accepted-anchor / first-token within N seconds â†’ cancel, full-resend recovery on the same account. **(R3)** reasoning-replay cache for *all* source protocols, so reasoning items survive a reattach.
- **Why:** the recurring "wedge" is SAME-account `store:false` anchor non-resumption on fresh-reattach full-resend (~31% of reattaches â€” see `sol-anchor-wedge-rootcause`). codex-lb's ownership guards (e0f3da3f/d846fd0a) did NOT fix it â†’ proves it is a *trim-vs-resend decision* bug, not an ownership bug. codex-lb decides trim/inject implicitly inline, which can't be reasoned about or unit-tested for the failure case; an explicit machine + watchdog turns an invisible 26â€“44s stall into a bounded, observable, recoverable event. R1 attacks the exact root cause: trimming assumes upstream persisted the prefix, but `store:false` means it explicitly did not â€” trimming there bets on state the account doesn't hold. R3 generalizes CLIProxyAPI's Claude-only reasoning replay so reattach doesn't drop chain-of-thought.
- **Trade-off:** R1 costs more upstream tokens on `store:false` reattaches (full-resend is bigger than a trimmed request). We accept that cost to eliminate a real 31% failure. R3 costs memory to hold reasoning items.
- **Revisit if:** false-recovery rate (watchdog firing on legitimately-slow-but-fine turns) proves high; or the reasoning cache footprint forces an eviction policy.

### C1(a) â€” trim policy â†’ **hard "never trim store:false full-resend" DEFAULT, per-account opt-in seam OFF by default**
- **Why:** correctness-first â€” the wedge is a proven 31% failure, so the safe path must be the default. But a provider/account that genuinely *does* persist `store:false` context could reclaim the token savings, so keep a per-account `allow_store_false_trim` override rather than hardcoding â€” seam exists, defaults safe.
- **Trade-off:** the safe default leaves token savings on the table for any account that could have resumed. Accept until an account is *proven* to resume reliably.
- **Revisit if:** an account is observed resuming `store:false` anchors reliably AND the extra full-resend token cost is material â†’ enable its override.

### C1(b) â€” watchdog timeout N â†’ **MVP fixed + configurable (~30s conservative), documented path to adaptive**
- **Why:** a fixed conservative N already converts the stall into a bounded event with zero new infra. Adaptive (N = per-account p95 TTFT Ã— factor, with floor/ceiling) is strictly better but needs the TTFT-histogram telemetry first â€” YAGNI for MVP. Default leans high because Sol first-token can legitimately be slow; erring high avoids false recoveries.
- **Trade-off:** fixed N too low â†’ false recoveries (wasted full-resends) on slow-but-healthy turns; too high â†’ wedge lingers longer before recovery. Conservative default biases toward not-false-recovering.
- **Revisit if:** TTFT telemetry lands (then switch to adaptive), or observed wedge-recovery latency feels too slow at the fixed value.

## T2 â€” Translator registry (the multi-provider spine) â†’ **`HashMap<(Format,Format),Translator>`; MVP = 3 entries, only `Anthropicâ†’Codex` is real**

## X1 â€” Testing & latency are first-class MVP deliverables (not follow-ups)
The spec mandates, shipped WITH the MVP: **(1)** an e2e harness with scriptable **mock upstreams** (fake OpenAI-Responses + fake Anthropic) that can emit exact SSE sequences, 429s, mid-stream stalls, and `store:false` non-resumption; **(2)** **golden replay tests** for the `Anthropicâ†’Codex` streaming translator (captured real event fixtures â†’ assert semantic equivalence, no buffering); **(3)** a **continuity wedge-regression e2e** that reproduces the store:false reattach and asserts R1/R2/R3 (never-trim, watchdog-recovers, reasoning-replayed); **(4)** a **latency-regression gate** in CI asserting PolyFlare's *own* added latency (`total âˆ’ upstream`) stays under a small threshold, plus time-to-first-translated-event. - **Why:** the two reasons PolyFlare exists are correctness (kill the wedge) and low latency â€” precisely the properties unit tests miss and only e2e/latency harnesses catch. The slowness saga *proved* codex-lb's own gate overhead was ~0 (â‰ˆ2ms/23k); PolyFlare must prove that continuously via a CI gate, not assert it by vibe. The wedge survived in codex-lb because there was no e2e reproduction of it â€” the wedge-regression test is what makes C1 actually stick.
- **Trade-off:** mock-upstream harness + fixtures are upfront cost before first ship. Accept â€” they are the regression floor for the two bugs the rebuild targets.
- **Revisit if:** fixtures drift from real upstream wire behavior (refresh from live captures); the latency threshold needs per-provider tuning.

## S3 â€” Selector / pool / policy â†’ **one default strategy behind a `Selector` trait; strategy expansion is test-and-see**
`Selector::pick(pool, ctx) â†’ Account`. MVP impl = codex-lb's **relative-availability + burn/preserve** scoring (its `logic.py` is pure math â†’ direct port + parity tests), wrapped by session-affinity. The other 7 codex-lb strategies (incl. force-finish / burn-a-near-max-account) are additive trait impls, added **only when empirical testing/live data justifies** â€” the user's explicit call ("test and see"), and precisely what the X1 harness exists to measure.
- **The load-bearing rule â€” selection ordering:** `(1) Continuity ownership â†’ (2) session affinity â†’ (3) availability scoring â†’ (4) health/cooldown gate`. Continuity ownership is a **hard pre-filter, not a scoring input**: an anchored conversation MUST return to its `owning_account` (or enter Recover). If availability scoring outranked continuity, the selector would route an anchored conversation to a "more available" cold account that doesn't hold its anchor â†’ **re-creates the wedge**. This is the rule that binds S3 to C1.
- **Health/cooldown** per-account-**per-model** (CLIProxyAPI `NextRetryAfter`/`Exceeded`/`blockReason` fused with codex-lb health tiers): an account rate-limited for `gpt-5.6-sol` can still serve `luna`; excluded accounts actually leave the loop.
- **Why one default:** better-ccflare proves 1 strategy is workable; codex-lb's 8 accreted. Adding strategies without pool-behavior data is speculation â€” defer to measurement.
- **Trade-off:** MVP may under-serve a routing scenario (e.g. force-finish burn) until we add its impl. Accept â€” additive, no core change.
- **Revisit if:** e2e/live data shows the default mis-routing (idle-account decay, a near-max account we want to force-finish, priority-drain) â†’ add the specific `Selector` impl.

### S3(a) â€” field proof + hardening: affinity is never ownership, disagreements never terminal, all affinity state TTL'd (2026-07-22 codex-lb incident)
A codex-lb production incident (`docs/incidents/2026-07-22-codex-lb-continuity-conflict.md`) proved S3's ordering is load-bearing: codex-lb let a TTL-less sticky-affinity row (account B, 9 days stale) contradict the resolved anchor owner (account A) and turned the disagreement into a **terminal 503 the client's verbatim retry could never escape** â€” 120+ consecutive failures, remedied only by hand-deleting live DB rows. PolyFlare's existing structure already forbids both failure modes; this addendum pins them with tests and closes the one shared gap:
- **(1) Affinity can never veto ownership** â€” `CodexContinuity::prepare` resolves the anchor map FIRST; the session row's `owning_account_id` only fills in when no anchor resolves, and the next `record_completion`/`record_recovery` overwrites a stale row. There is deliberately NO conflict-error path. Locked by unit tests.
- **(2) Disagreements resolve, never error-loop** â€” a pinned-but-ineligible owner is `RouteDecision::Recover` (strip anchor â†’ reselect â†’ full resend â†’ `record_recovery` re-homes; the TA6(b) path â€” transport findings proved HTTP-SSE full resend cheap + wedge-immune), never a client-visible error. The bar for any future client-visible error: the client's NATURAL retry must converge (Codex CLI retries the same logical turn verbatim). Locked by the e2e fixture: a repeated identical follow-up must never fail the same way twice.
- **(3) All affinity state gets a TTL** â€” `run_retention_pass` prunes `continuity_sessions` (by `last_activity_at`) + `continuity_anchors` (by `created_at`) at a fixed always-on 30-day TTL (NOT a runtime setting â€” deletion is proven safe at any moment, so the exact value is uncritical; a disabled-by-default knob would recreate codex-lb's no-TTL trap). Ownership survives an affinity wipe by construction (anchor map is the hard source) â€” locked by a mid-conversation `DELETE FROM continuity_sessions` e2e scenario.
- **(4) Resolution observability** â€” one content-free `continuity_owner_resolution` log line per resolution (anchor owner / session owner / which won / `stale_affinity` flag); this line is what made the codex-lb incident diagnosable in minutes.
- **Regression fixture:** `crates/polyflare-server/tests/continuity_owner_conflict.rs` encodes the exact incident state-sequence (anchored on A, stale affinity â†’ B, follow-up carries the A-owned anchor).
- **Revisit if:** a second HARD ownership source ever appears (e.g. a cluster-mode runtime map alongside the store) â†’ the split-brain rule in (2) applies as written: Recover, never a terminal error.

## E4 â€” Executors + transport â†’ **`Executor` trait, transport below it; Codex = WS-first + fingerprint laundering; Anthropic = HTTP + ccflare rate-limit semantics**
`Executor::execute(PreparedReq, Account) â†’ ResponseStream`. Transport (WS / SSE / HTTP) sits **below** the executor behind a common streaming interface, so continuity (C1) and translation (T2) live above and are transport-agnostic. `polyflare-codex`: WS-first (`supports_websockets=true`, SSE fallback on 426 â€” port it, don't invert), `codex_cli_rs` UA + `x-stainless-*` stripping + native-vs-SDK detection keyed only on UA/originator (never replayable turn-state), plugs the Codex `Continuity` impl. `polyflare-anthropic`: HTTP + Anthropic OAuth + ccflare rate-limit header semantics (`out_of_credits`/`extra_usage`/`529`/24h-clamp) as a typed module, no-op continuity.
- **The Rust-only win:** `rustls`/BoringSSL give real ClientHello/JA3/ALPN control. codex-lb faked the UA but ran Python's TLS stack, so its handshake never matched the real Codex CLI â€” one of the confirmed egress gaps (`codex-fingerprint-gaps`: 10 gaps, 5 HIGH turn-state identity headers). PolyFlare can close it. This is a concrete reason to rebuild rather than refactor.

### E4(a) â€” fingerprint fidelity target â†’ **(b) byte-identical uTLS-style parity, chased continuously**
User chose max fidelity over my recommended "(a) match-the-shape". Kept affordable by the key enabler below.
- **Why (b) is tractable here (the enabler):** the impersonation target is **itself a Rust/reqwest client (`codex-rs`), not a browser.** uTLS treadmills exist to chase *Chrome*; here byte-parity likely reduces to **pinning PolyFlare's TLS/HTTP dependency stack to codex-rs's** (same `reqwest` + TLS backend + `h2` versions â†’ matching ClientHello, cipher/extension order, GREASE, HTTP/2 SETTINGS for free). Sustained by a **capture-fixture + CI parity-diff gate**: capture the real CLI's egress fingerprint per Codex release as a versioned fixture, assert PolyFlare byte-matches. Treadmill â†’ version-bump + CI diff, not reverse-engineering.
- **Trade-off:** version-coupling to codex-rs's HTTP/TLS deps; a parity-diff gate to maintain; if codex-rs's chosen TLS backend isn't the one that gives us ClientHello control (e.g. it uses `native-tls`), we may need BoringSSL (`boring`/`tokio-boring`) + a ClientHello customization layer instead of pure `rustls`.
- **MUST verify at implementation time:** codex-rs's actual TLS backend + reqwest/h2 versions (authoritative source: `codex-model-metadata-sources` â†’ client.rs + Cargo.toml). The whole affordability argument for (b) rests on this.
- **Revisit if:** codex-rs switches TLS stack, or the version-coupling/parity-gate upkeep outweighs the fidelity benefit â†’ fall back to E4(a)-option-(a) "match the shape".

### SE5(a) â€” token encryption at rest â†’ **Rust-native AEAD (XChaCha20-Poly1305), NOT Fernet-compatible** *(reversed my initial recommendation â€” user was right)*
- **Why (the correction):** I had conflated "zero re-auth" with "keep Fernet forever" â€” they are **separable**. Zero re-auth only requires the one-time **import tool** to decrypt the old Python Fernet blobs once (it already holds the key; `migrate_oauth_usage.py` already touches these blobs) and re-encrypt native on the way in. So Rust-native delivers BOTH a clean permanent codebase AND zero re-auth. "Faster" (single-pass AEAD + AES-NI vs CBC+HMAC) is real but **off the hot path** (tokens decrypt rarely, cached in memory) â†’ ~zero latency impact; decide on cleanliness. For a from-scratch rebuild escaping codex-lb's accretion, carrying a legacy Python crypto format indefinitely is the wrong trade.
- **Why XChaCha20-Poly1305 specifically:** 192-bit nonce â†’ safe with a random nonce per blob (no AES-GCM 96-bit nonce-reuse fragility to manage). External key file retained (operator-familiar key mgmt; own key, not the Fernet one).
- **Trade-off:** a one-time re-encrypt step in the importer (trivial â€” the script already decrypts Fernet). PolyFlare can't read raw codex-lb blobs without that import â€” acceptable for a rebuild, not a drop-in.
- **Revisit if:** n/a (settled). Nonce-management concern is pre-empted by the XChaCha choice.

## TA6 â€” Trusted-access / capability-gated account routing ("Trusted Access for Cyber" + Claude equivalent) â†’ **reactive detection + eligibility-filtered retry; continuity ownership wins**
Grounded in codex-lb's *actual* mechanism (verified 2026-07-14): it is NOT a generic entitlement/tier system â€” it's a single operator-set boolean `Account.security_work_authorized`. PolyFlare's design:
- **Account model** (`polyflare-store`): a small per-account **capability/entitlement set**, **operator-set** â€” NOT derived from OAuth claims (confirmed: no token/ID claim carries trust/cyber info; only `plan_type`/`seat_type`/`workspace_id` come from claims). Generalize codex-lb's single bool into a capability set ONLY because PolyFlare is dual-provider (two concrete members: Codex `SecurityWorkAuthorized`, Anthropic equivalent) â€” a justified generalization, not speculation.
- **Trigger is REACTIVE, not proactive** (the load-bearing, non-obvious point): NO client header / model name / endpoint / token scope predicts the need. OpenAI rejects the in-flight request (error code `security_work_authorization_required`, OR a message containing ALL of "flagged for possible cybersecurity risk" + "authorized for security work" + "chatgpt.com/cyber"), and only THEN does routing retry on an eligible account. So the **Executor classifies the provider rejection into a neutral `NeedsCapability(cap)` error**; the retry loop reacts.
- **Selector eligibility filter** (extends S3): when a retry requires capability X, only accounts holding X are eligible â€” a HARD filter, like continuity ownership.
- **Degrade path â€” preserve EXACTLY** (security-sensitive; codex-lb rule: "trusted-access routing must degrade only along the documented path"): (1) upstream rejects on current acct â†’ (2) if request is **UNPINNED** and the erroring acct isn't itself authorized â†’ exclude it, re-select requiring the cap; (3) eligible acct found â†’ retry on it; (4) **NO** eligible acct â†’ warn + fall back to normal unfiltered failover (do NOT hard-fail); (5) **PINNED requests (file-owned / `previous_response_id`-chained / last attempt) are NEVER moved** â†’ surface the original error to the client unchanged.
- **This VALIDATES S3's ordering:** pinned == continuity-owned, and continuity ownership is the top hard pre-filter â†’ it **wins** over the trusted-access retry by construction. Moving a pinned request to a trusted account would re-wedge; the two rules compose correctly for free.
- **Claude equivalent = identical architecture.** The `polyflare-anthropic` executor supplies (a) the per-account flag (Anthropic approved-org / elevated dual-use entitlement) and (b) the provider-specific rejection detection. Anthropic's exact error signal + whether it exposes an org "approved" flag â†’ **VERIFY when building `polyflare-anthropic` (M4)**; the neutral `NeedsCapability`/eligibility machinery is shared.
- **Scope:** M2/M3 (store capability field + Selector eligibility + retry orchestration); Anthropic specifics at M4. NOT M1.
- **Trade-off:** reactive-only means a request can "waste" one upstream round-trip on a non-authorized account before retrying â€” accept (matches codex-lb; unavoidable without a proactive client signal).
- **Revisit if:** a provider exposes a proactive capability signal (token scope / model tag) â†’ pre-filter instead of react; or a real multi-capability need emerges â†’ the capability-set model already accommodates it.
- **codex-lb source refs** (for the M2/M3 port): column `app/db/models.py` `security_work_authorized`; selector `app/modules/proxy/load_balancer.py:337-397`; detection `app/modules/proxy/service.py:867-883,2352-2383`; retry orchestration `_service/streaming/retry.py`, `_service/http_bridge/request_submit.py`, `_service/websocket/mixin.py`; normative spec `openspec/changes/archive/2026-07-12-add-security-work-account-routing/`.

### TA6(a) â€” Proactive cyber-binding (improvement over codex-lb's reactive-only) â†’ **bind at CREDENTIAL/PROFILE scope, not session scope; make the session requirement sticky**
codex-lb is purely reactive (waste one upstream round-trip, then retry). PolyFlare adds proactive binding so cyber-scoped work hits a trusted account from turn 1. A request's `required_capabilities` resolves by precedence:
1. **Request-level** â€” a header (`X-PolyFlare-Capability: security_work`) or a model alias (`â€¦-cyber`). The Codex/Claude CLI can carry this via its provider `http_headers` config, so a dedicated "cyber" CLI profile tags every request. Most specific.
2. **Credential / client-profile-level** â€” dashboard-managed: "API key `pf_cyber_â€¦` / named client profile always requires `security_work`." **This is the answer to the session-naming problem** (below): known on request 1, human-nameable â†’ true from-start binding.
3. **Session sticky-flag** â€” once a session is determined to need cyber (proactively OR after a reactive rejection), persist `required_capabilities` on the session/continuity state so ALL later turns pre-filter to trusted accounts. Concrete improvement over codex-lb, which re-reacts every request.
4. **Reactive fallback** â€” the TA6 upstream-rejection path, unchanged, as the safety net.
- **Why credential-scope for from-start, not session-scope:** at the proxy, PolyFlare only sees an opaque `session_id` (derived from headers / content-hash), never a human name â€” the operator can't pick "the session about X" before it exists, and by the time a session appears in the dashboard its first request already left. API keys / client profiles ARE human-nameable and present on request 1 â†’ the correct hook for "this whole stream of work is cyber."
- **Dashboard session-grain pin (secondary tool):** to pin an already-running conversation you forgot to pre-declare, the dashboard identifies live sessions by **content-preview + model + recency + current account** (better-ccflare's conversation viewer), not a name. Works turn 2+, not turn 1.
- **End-to-end:** cyber CLI profile (header / cyber API key) â†’ PolyFlare resolves `required_capabilities={security_work}` â†’ Selector pre-filters to trusted pool from turn 1 â†’ session sticky-flag keeps it there. Same shape for Claude via the Anthropic executor.
- **Scope:** M2/M3 (capability resolution + selector eligibility + session sticky-flag + per-API-key capability config); dashboard UI is L5; Anthropic specifics M4.
- **Revisit if:** clients can't set headers/aliases in practice â†’ lean on credential-scope; or a session-label header becomes standard â†’ could name sessions directly.

### TA6(b) â€” pinned requests DO move on a hard capability rejection â†’ **REVERSES TA6 point 5** (user-directed, 2026-07-17; security-sensitive, explicitly signed off)
**Field evidence:** in codex-lb, a session that hit the cyber requirement while sticky/pinned got **stuck** â€” it never moved to a cyber-capable account, it just kept failing on the pinned owner. That is exactly what TA6 point 5 ("PINNED requests are NEVER moved â†’ surface the error unchanged") prescribes. The incident is direct evidence that rule is wrong.
- **Why TA6.5's premise is now obsolete:** it rested on *"moving a pinned request would re-wedge"* â€” i.e. the belief that dropping the anchor and re-sending is expensive/dangerous. The live transport measurements ([[handover-prefill-experiment]], `TRANSPORT-FINDINGS-2026-07-17.md`) **disproved** that: an HTTP-SSE full resend to another account is cheap and wedge-immune (~free, only an org-cache miss), and on WS it is a fast catchable error, not a hang. So "never move" now trades a **cheap resend** for a **guaranteed client-facing failure** (the owner rejects *every* cyber turn) â€” a bad trade. The wedge that justified the rule only ever lived on a narrow WS-anchor path, not here.
- **The revision:** a pinned/continuity-owned session that hits a **hard capability rejection** (cyber) its owner can't satisfy **DOES move** to a capability-holding account: strip the (worthless-for-this-turn) anchor â†’ re-select from accounts holding the cap â†’ full resend â†’ `record_recovery` re-homes session ownership to the new account â†’ stamp the **session sticky-flag** (TA6(a).3) so every later turn pre-filters to capable accounts (pay the reject-and-move cost ONCE). Capability-holding accounts are a **superset** (they also serve non-cyber), so staying there after the move costs nothing.
- **Not a security bypass â€” it falls the RIGHT way:** moving *to a cyber-authorized account* routes to the account that IS authorized; the invariant "never serve cyber work on a non-authorized account" is preserved. This is the opposite of falling open.
- **Security floor â€” REVISES codex-lb degrade step 4 for the cyber case (user chose STRICT):** when a session is sticky-cyber and NO capable account is available (all rate-limited/down), **fail with a clear "no authorized account available" error â€” NEVER fall back to unfiltered failover onto a non-authorized account.** codex-lb's documented degrade step 4 (warn + unfiltered failover) is correct for a *capacity* miss but wrong for a *capability* miss: on a cyber-flagged request an unauthorized account only rejects anyway, so falling back buys a muddier failure, not success. Cyber work waits or fails; it never lands on a non-authorized account.
- **Dedicated "cyber pool" (proactive, user-requested):** tag a **pool** with a pool-level `required_capability = security_work`. Any request routed there (`/cyber/responses` or a `pf_cyber_â€¦` key) auto-resolves `required_capabilities={security_work}` from turn 1, and the selector pre-filters to the pool's capability-holding accounts. **Self-enforcing:** the pool tag declares the requirement, the per-account flag declares who satisfies it â€” a non-cyber account accidentally placed in the pool is still excluded by the capability filter. This is TA6(a)'s proactive precedence-1/2 hook expressed as a pool. So both paths exist: tag-a-pool-cyber for known-cyber work (zero wasted round-trips), and reactive auto-move+sticky for sessions that only reveal the need mid-stream.
- **Dependency + sequencing:** detecting the rejection needs the upstream **error-code string** in the failure signal â€” the SAME enabler A6 (quota) and A7 (auth-writeback) need (`FailureSignal`/`ExecError::UpstreamStatus` carries only `{status, retry_after}` today; see PORTING Phase A audit). So cyber-move rides the exact A6/A7 work and is **gated behind M5a** (which is currently reshaping that `ExecError` path â€” concurrent edits would collide). One enabler unlocks three routing behaviors.
- **Revisit if:** a capability rejection ever proves NOT reliably classifiable from the error code/body â†’ fall back to the reactive-only detection TA6 already documents (unchanged as the safety net).

## M2-GATE1 â€” trait seam shapes are PROVISIONAL, not frozen (reshape at M2 kickoff) â€” from M1 whole-branch review
The M1 spec called the five trait signatures "fixed seams." The M1 whole-branch review (opus) flags that three are the wrong shape and will force changes once M2/M3 build real logic on them. **Decision: drop the "frozen" label for these three; the FIRST M2 design gate is reshaping them with real requirements in hand.** `Translator` and `Executor` are well-shaped (async+Result / pure transform) and hold.
- **`Selector::pick(&self, &[Account], &RequestCtx) -> Option<&Account>`** â€” inputs too thin. Real selection scores on live per-account runtime state (in-flight count, rate-limit remaining, cooldown, health) + continuity ownership + TA6 capability eligibility â€” none reachable from static `Account` (id/base_url/token) + `RequestCtx{session_id}`. `&[Account]` also forces a contiguous Vec pool (rules out a map-keyed pool). M2 must enrich the inputs before code depends on it.
- **`Continuity::prepare(&self, req, ctx) -> PreparedRequest`** â€” sync, infallible, store-less. The M3 continuity state machine (anchor/trim/resend â€” the wedge-prone core, [[sol-anchor-wedge-rootcause]]) MUST read/write persisted session state (async sqlx), can fail, and per TA6(a) stamps a session sticky-flag. Almost certainly becomes `async fn prepare(...) -> Result<PreparedRequest, ContinuityError>` with a session-state handle. Highest-confidence change.
- **`Coordinator::admit(&self, ctx) -> bool`** â€” admission + session-ownership typically must `await` (acquire permit / claim ownership) and return an RAII guard whose `Drop` releases the slot. `-> bool` expresses neither; likely `async fn admit(...) -> Result<AdmissionGuard, Rejected>`.
- **Why provisional, not reshaped now:** nothing implements these in M1, and shaping async/Result/guard signatures without M2's actual requirements is speculation (YAGNI). Reshape at M2 design â€” cheap (in-workspace, zero dependents today).
- **M4 note:** `Translator::translate_response_event(&self, Value) -> Value` is stateless 1-in-1-out; a real non-identity cross-format streaming translator (Anthropicâ†”OpenAI) often needs `&mut self` state + 1â†’N output. Revisit when the seam meets its first non-identity translator.
- **Related M2 follow-ups from the review:** (1) add an idle/read-timeout **watchdog** on the response stream â€” only `connect_timeout` is set today, so a stalled upstream hangs forever (aligns with codex-lb watchdog follow-ups); (2) reinstate upstream-error-**body capture** in the executor â€” TA6 reactive trusted-access detection classifies that body; (3) map the client-facing **502 to a generic body** (don't forward reqwest `Display`, which can carry the upstream URL).
