# PolyFlare — Design Decisions (brainstorming log)

Running record of locked decisions. Updated as the design conversation proceeds.

**Convention:** every entry carries a **reasoning trace** — *Why* (the rationale), *Trade-off* (what the choice costs), and *Revisit if* (the concrete signal that should reopen it) — so a future reader can reconstruct the decision, not just read the verdict.

## Name → **PolyFlare**
Descriptive-compound in the `ccflare` / `codex-lb` convention: **poly-** (multi-provider — the differentiator vs the single-provider ancestors) + **-flare** (the ccflare/Cloudflare edge-proxy lineage). Binary `polyflare` (subcommands `serve`, `accounts`, …); workspace crates `polyflare-core` / `-codex` / `-anthropic` / `-store` / `-server`.
- Collision check (2026-07-14): no LLM/CLI/proxy-space owner. GitHub `polyflare` repos are unrelated personal projects (a WGPU lens-flare graphics thesis) → the **org** name is free. `polyflare.org` is a taken (unrelated crypto tool); `.dev`/`.rs`/`.sh` likely open.

## Q1 — Center of gravity → **B: Dual-upstream, provider-neutral core**
The core is protocol-neutral from day one. A **translator registry** (`Format` enum + `(from,to) → Translator` map, borrowed from CLIProxyAPI) is the spine; Codex and Anthropic are both **first-class backends** in the MVP, and any supported client protocol can route to either pool. codex-lb's Codex depth (fingerprint, quota-routing, continuity) becomes the Codex *backend implementation*, not the whole system.
- Implication: bigger MVP, but adding a provider is additive (register translators + an executor), never a rewrite.
- Rejected: (A) Codex-first (would need refactor to generalize later), (C) translator-first (defers the depth that's the actual value).

## Q2 — Deployment/coordination → **A+: single-binary with cluster-shaped seams**
One tokio process, one embedded store, all state in-process. **Deletes codex-lb's entire multi-replica stack** (the ~8.4k-LOC bridge owner-forwarding subsystem, cap-partitioning, leader election, cache-invalidation poller, shared-Fernet key, advertise URLs) — all of which existed only because upstream targets team/k8s deployments the user doesn't run.
- Session-ownership + admission sit behind a `Coordinator` **trait** (in-process impl for v1) so a future distributed adapter is additive, not a rewrite. Revisit only if HA/multi-machine becomes a real need.
- "Multiple PCs" = multiple *clients* → still ONE LB (network-reachable via LAN/Tailscale); a single multi-threaded tokio process (no GIL) handles heavy multi-client concurrency far better than Python codex-lb would. Cluster is only for HA of the LB box itself.
- Same posture as better-ccflare / CLIProxyAPI (single local process); codex-lb was the outlier.

## Q3 — MVP line → **(i) Lean core, L0–L4**
First shippable = working dual-pool gateway: neutral core + translator registry (L0), Codex + Anthropic executors incl. the continuity state-machine + watchdog wedge-fix (L1), selector/pool/policy (L2), OpenAI-Responses + Anthropic-Messages ingress + Anthropic→Codex translation (L3), thin observability + API-key auth + account OAuth/import (L4). **No dashboard in MVP** — it's the clean additive L5 (reads the same REST API); codex-lb remains the operator UI until then.

## C1 — Continuity engine (the wedge fix) → **explicit per-conversation state machine + watchdog**
Model each conversation as an explicit state machine (`Fresh → Anchored → Reattaching → {Anchored | Recover}`) owned in `polyflare-store` as `{anchor_response_id, owning_account, last_good_turn, reasoning_cache_ref, state}`, behind a `Continuity` trait (`prepare(req,state)→PreparedReq` / `observe(resp)→Transition`). Three rules: **(R1)** never trim a `store:false` full-resend to an anchor the upstream isn't guaranteed to hold; trim only when the client kept `store:true` history. **(R2)** every `Reattaching` gets an async watchdog — no accepted-anchor / first-token within N seconds → cancel, full-resend recovery on the same account. **(R3)** reasoning-replay cache for *all* source protocols, so reasoning items survive a reattach.
- **Why:** the recurring "wedge" is SAME-account `store:false` anchor non-resumption on fresh-reattach full-resend (~31% of reattaches — see `sol-anchor-wedge-rootcause`). codex-lb's ownership guards (e0f3da3f/d846fd0a) did NOT fix it → proves it is a *trim-vs-resend decision* bug, not an ownership bug. codex-lb decides trim/inject implicitly inline, which can't be reasoned about or unit-tested for the failure case; an explicit machine + watchdog turns an invisible 26–44s stall into a bounded, observable, recoverable event. R1 attacks the exact root cause: trimming assumes upstream persisted the prefix, but `store:false` means it explicitly did not — trimming there bets on state the account doesn't hold. R3 generalizes CLIProxyAPI's Claude-only reasoning replay so reattach doesn't drop chain-of-thought.
- **Trade-off:** R1 costs more upstream tokens on `store:false` reattaches (full-resend is bigger than a trimmed request). We accept that cost to eliminate a real 31% failure. R3 costs memory to hold reasoning items.
- **Revisit if:** false-recovery rate (watchdog firing on legitimately-slow-but-fine turns) proves high; or the reasoning cache footprint forces an eviction policy.

### C1(a) — trim policy → **hard "never trim store:false full-resend" DEFAULT, per-account opt-in seam OFF by default**
- **Why:** correctness-first — the wedge is a proven 31% failure, so the safe path must be the default. But a provider/account that genuinely *does* persist `store:false` context could reclaim the token savings, so keep a per-account `allow_store_false_trim` override rather than hardcoding — seam exists, defaults safe.
- **Trade-off:** the safe default leaves token savings on the table for any account that could have resumed. Accept until an account is *proven* to resume reliably.
- **Revisit if:** an account is observed resuming `store:false` anchors reliably AND the extra full-resend token cost is material → enable its override.

### C1(b) — watchdog timeout N → **MVP fixed + configurable (~30s conservative), documented path to adaptive**
- **Why:** a fixed conservative N already converts the stall into a bounded event with zero new infra. Adaptive (N = per-account p95 TTFT × factor, with floor/ceiling) is strictly better but needs the TTFT-histogram telemetry first — YAGNI for MVP. Default leans high because Sol first-token can legitimately be slow; erring high avoids false recoveries.
- **Trade-off:** fixed N too low → false recoveries (wasted full-resends) on slow-but-healthy turns; too high → wedge lingers longer before recovery. Conservative default biases toward not-false-recovering.
- **Revisit if:** TTFT telemetry lands (then switch to adaptive), or observed wedge-recovery latency feels too slow at the fixed value.

## T2 — Translator registry (the multi-provider spine) → **`HashMap<(Format,Format),Translator>`; MVP = 3 entries, only `Anthropic→Codex` is real**
Registry keyed `(from,to)`; same-format pairs are **identity** (zero cost). MVP: identity `OpenAIResponses↔` (real Codex CLI → Codex pool) + identity `AnthropicMessages↔` (Claude Code → Anthropic pool) + the one cross-translator **`AnthropicMessages → OpenAIResponses`** (Claude-format client / subagent → Codex pool). Inverse (`Codex→Anthropic`) and Gemini/ChatCompletions deferred.
- **Why:** the registry is what generalizes beyond Codex (Q1=B). Identity-for-same-format keeps the native Codex and native Claude paths zero-overhead — translation only pays when a request actually crosses providers. Ship only `Anthropic→Codex` because it's the sole stated consumer ("Claude CLI for local code routed to Sol", subagent routing); the inverse has no current user (YAGNI). Event-by-event SSE stream mapping (`message_start`/`content_block_delta`/`message_delta` ↔ `response.output_text.delta`/`response.completed`) is the real complexity — concentrating it in ONE translator makes it testable.
- **Trade-off:** a Codex-format client can't reach the Claude pool in MVP. Purely additive later (register 1 translator, maybe 1 executor) — never a core rewrite.
- **Revisit if:** a real consumer needs `Codex→Anthropic` or a new provider format appears.

## X1 — Testing & latency are first-class MVP deliverables (not follow-ups)
The spec mandates, shipped WITH the MVP: **(1)** an e2e harness with scriptable **mock upstreams** (fake OpenAI-Responses + fake Anthropic) that can emit exact SSE sequences, 429s, mid-stream stalls, and `store:false` non-resumption; **(2)** **golden replay tests** for the `Anthropic→Codex` streaming translator (captured real event fixtures → assert semantic equivalence, no buffering); **(3)** a **continuity wedge-regression e2e** that reproduces the store:false reattach and asserts R1/R2/R3 (never-trim, watchdog-recovers, reasoning-replayed); **(4)** a **latency-regression gate** in CI asserting PolyFlare's *own* added latency (`total − upstream`) stays under a small threshold, plus time-to-first-translated-event. Port codex-lb's 6-phase latency model as the phase breakdown.
- **Why:** the two reasons PolyFlare exists are correctness (kill the wedge) and low latency — precisely the properties unit tests miss and only e2e/latency harnesses catch. The slowness saga *proved* codex-lb's own gate overhead was ~0 (≈2ms/23k); PolyFlare must prove that continuously via a CI gate, not assert it by vibe. The wedge survived in codex-lb because there was no e2e reproduction of it — the wedge-regression test is what makes C1 actually stick.
- **Trade-off:** mock-upstream harness + fixtures are upfront cost before first ship. Accept — they are the regression floor for the two bugs the rebuild targets.
- **Revisit if:** fixtures drift from real upstream wire behavior (refresh from live captures); the latency threshold needs per-provider tuning.

## S3 — Selector / pool / policy → **one default strategy behind a `Selector` trait; strategy expansion is test-and-see**
`Selector::pick(pool, ctx) → Account`. MVP impl = codex-lb's **relative-availability + burn/preserve** scoring (its `logic.py` is pure math → direct port + parity tests), wrapped by session-affinity. The other 7 codex-lb strategies (incl. force-finish / burn-a-near-max-account) are additive trait impls, added **only when empirical testing/live data justifies** — the user's explicit call ("test and see"), and precisely what the X1 harness exists to measure.
- **The load-bearing rule — selection ordering:** `(1) Continuity ownership → (2) session affinity → (3) availability scoring → (4) health/cooldown gate`. Continuity ownership is a **hard pre-filter, not a scoring input**: an anchored conversation MUST return to its `owning_account` (or enter Recover). If availability scoring outranked continuity, the selector would route an anchored conversation to a "more available" cold account that doesn't hold its anchor → **re-creates the wedge**. This is the rule that binds S3 to C1.
- **Health/cooldown** per-account-**per-model** (CLIProxyAPI `NextRetryAfter`/`Exceeded`/`blockReason` fused with codex-lb health tiers): an account rate-limited for `gpt-5.6-sol` can still serve `luna`; excluded accounts actually leave the loop.
- **Why one default:** better-ccflare proves 1 strategy is workable; codex-lb's 8 accreted. Adding strategies without pool-behavior data is speculation — defer to measurement.
- **Trade-off:** MVP may under-serve a routing scenario (e.g. force-finish burn) until we add its impl. Accept — additive, no core change.
- **Revisit if:** e2e/live data shows the default mis-routing (idle-account decay, a near-max account we want to force-finish, priority-drain) → add the specific `Selector` impl.

## E4 — Executors + transport → **`Executor` trait, transport below it; Codex = WS-first + fingerprint laundering; Anthropic = HTTP + ccflare rate-limit semantics**
`Executor::execute(PreparedReq, Account) → ResponseStream`. Transport (WS / SSE / HTTP) sits **below** the executor behind a common streaming interface, so continuity (C1) and translation (T2) live above and are transport-agnostic. `polyflare-codex`: WS-first (`supports_websockets=true`, SSE fallback on 426 — port it, don't invert), `codex_cli_rs` UA + `x-stainless-*` stripping + native-vs-SDK detection keyed only on UA/originator (never replayable turn-state), plugs the Codex `Continuity` impl. `polyflare-anthropic`: HTTP + Anthropic OAuth + ccflare rate-limit header semantics (`out_of_credits`/`extra_usage`/`529`/24h-clamp) as a typed module, no-op continuity.
- **The Rust-only win:** `rustls`/BoringSSL give real ClientHello/JA3/ALPN control. codex-lb faked the UA but ran Python's TLS stack, so its handshake never matched the real Codex CLI — one of the confirmed egress gaps (`codex-fingerprint-gaps`: 10 gaps, 5 HIGH turn-state identity headers). PolyFlare can close it. This is a concrete reason to rebuild rather than refactor.

### E4(a) — fingerprint fidelity target → **(b) byte-identical uTLS-style parity, chased continuously**
User chose max fidelity over my recommended "(a) match-the-shape". Kept affordable by the key enabler below.
- **Why (b) is tractable here (the enabler):** the impersonation target is **itself a Rust/reqwest client (`codex-rs`), not a browser.** uTLS treadmills exist to chase *Chrome*; here byte-parity likely reduces to **pinning PolyFlare's TLS/HTTP dependency stack to codex-rs's** (same `reqwest` + TLS backend + `h2` versions → matching ClientHello, cipher/extension order, GREASE, HTTP/2 SETTINGS for free). Sustained by a **capture-fixture + CI parity-diff gate**: capture the real CLI's egress fingerprint per Codex release as a versioned fixture, assert PolyFlare byte-matches. Treadmill → version-bump + CI diff, not reverse-engineering.
- **Trade-off:** version-coupling to codex-rs's HTTP/TLS deps; a parity-diff gate to maintain; if codex-rs's chosen TLS backend isn't the one that gives us ClientHello control (e.g. it uses `native-tls`), we may need BoringSSL (`boring`/`tokio-boring`) + a ClientHello customization layer instead of pure `rustls`.
- **MUST verify at implementation time:** codex-rs's actual TLS backend + reqwest/h2 versions (authoritative source: `codex-model-metadata-sources` → client.rs + Cargo.toml). The whole affordability argument for (b) rests on this.
- **Revisit if:** codex-rs switches TLS stack, or the version-coupling/parity-gate upkeep outweighs the fidelity benefit → fall back to E4(a)-option-(a) "match the shape".

## SE5 — Server edge (ingress / auth / store / crypto / observability / coordinator seam)
`polyflare-server` (axum): two ingress decode paths (OpenAI-Responses for real Codex CLI; Anthropic-Messages for Claude-format clients) → neutral core; API-key auth (admin + client keys). `polyflare-store`: **SQLite via `sqlx`**, clean schema day one (port codex-lb's *final* tables only — accounts, usage, request_logs, continuity session-state — **drop all 176 Alembic migrations**). Observability **thin**: 6-phase latency model + small Prometheus set + request logs; expose 3–4 decision numbers, not the 53-col/25-field sprawl. **Coordinator trait** seam (A+): session-ownership + admission behind a trait, in-process impl v1.

### SE5(a) — token encryption at rest → **Rust-native AEAD (XChaCha20-Poly1305), NOT Fernet-compatible** *(reversed my initial recommendation — user was right)*
- **Why (the correction):** I had conflated "zero re-auth" with "keep Fernet forever" — they are **separable**. Zero re-auth only requires the one-time **import tool** to decrypt the old Python Fernet blobs once (it already holds the key; `migrate_oauth_usage.py` already touches these blobs) and re-encrypt native on the way in. So Rust-native delivers BOTH a clean permanent codebase AND zero re-auth. "Faster" (single-pass AEAD + AES-NI vs CBC+HMAC) is real but **off the hot path** (tokens decrypt rarely, cached in memory) → ~zero latency impact; decide on cleanliness. For a from-scratch rebuild escaping codex-lb's accretion, carrying a legacy Python crypto format indefinitely is the wrong trade.
- **Why XChaCha20-Poly1305 specifically:** 192-bit nonce → safe with a random nonce per blob (no AES-GCM 96-bit nonce-reuse fragility to manage). External key file retained (operator-familiar key mgmt; own key, not the Fernet one).
- **Trade-off:** a one-time re-encrypt step in the importer (trivial — the script already decrypts Fernet). PolyFlare can't read raw codex-lb blobs without that import — acceptable for a rebuild, not a drop-in.
- **Revisit if:** n/a (settled). Nonce-management concern is pre-empted by the XChaCha choice.

## TA6 — Trusted-access / capability-gated account routing ("Trusted Access for Cyber" + Claude equivalent) → **reactive detection + eligibility-filtered retry; continuity ownership wins**
Grounded in codex-lb's *actual* mechanism (verified 2026-07-14): it is NOT a generic entitlement/tier system — it's a single operator-set boolean `Account.security_work_authorized`. PolyFlare's design:
- **Account model** (`polyflare-store`): a small per-account **capability/entitlement set**, **operator-set** — NOT derived from OAuth claims (confirmed: no token/ID claim carries trust/cyber info; only `plan_type`/`seat_type`/`workspace_id` come from claims). Generalize codex-lb's single bool into a capability set ONLY because PolyFlare is dual-provider (two concrete members: Codex `SecurityWorkAuthorized`, Anthropic equivalent) — a justified generalization, not speculation.
- **Trigger is REACTIVE, not proactive** (the load-bearing, non-obvious point): NO client header / model name / endpoint / token scope predicts the need. OpenAI rejects the in-flight request (error code `security_work_authorization_required`, OR a message containing ALL of "flagged for possible cybersecurity risk" + "authorized for security work" + "chatgpt.com/cyber"), and only THEN does routing retry on an eligible account. So the **Executor classifies the provider rejection into a neutral `NeedsCapability(cap)` error**; the retry loop reacts.
- **Selector eligibility filter** (extends S3): when a retry requires capability X, only accounts holding X are eligible — a HARD filter, like continuity ownership.
- **Degrade path — preserve EXACTLY** (security-sensitive; codex-lb rule: "trusted-access routing must degrade only along the documented path"): (1) upstream rejects on current acct → (2) if request is **UNPINNED** and the erroring acct isn't itself authorized → exclude it, re-select requiring the cap; (3) eligible acct found → retry on it; (4) **NO** eligible acct → warn + fall back to normal unfiltered failover (do NOT hard-fail); (5) **PINNED requests (file-owned / `previous_response_id`-chained / last attempt) are NEVER moved** → surface the original error to the client unchanged.
- **This VALIDATES S3's ordering:** pinned == continuity-owned, and continuity ownership is the top hard pre-filter → it **wins** over the trusted-access retry by construction. Moving a pinned request to a trusted account would re-wedge; the two rules compose correctly for free.
- **Claude equivalent = identical architecture.** The `polyflare-anthropic` executor supplies (a) the per-account flag (Anthropic approved-org / elevated dual-use entitlement) and (b) the provider-specific rejection detection. Anthropic's exact error signal + whether it exposes an org "approved" flag → **VERIFY when building `polyflare-anthropic` (M4)**; the neutral `NeedsCapability`/eligibility machinery is shared.
- **Scope:** M2/M3 (store capability field + Selector eligibility + retry orchestration); Anthropic specifics at M4. NOT M1.
- **Trade-off:** reactive-only means a request can "waste" one upstream round-trip on a non-authorized account before retrying — accept (matches codex-lb; unavoidable without a proactive client signal).
- **Revisit if:** a provider exposes a proactive capability signal (token scope / model tag) → pre-filter instead of react; or a real multi-capability need emerges → the capability-set model already accommodates it.
- **codex-lb source refs** (for the M2/M3 port): column `app/db/models.py` `security_work_authorized`; selector `app/modules/proxy/load_balancer.py:337-397`; detection `app/modules/proxy/service.py:867-883,2352-2383`; retry orchestration `_service/streaming/retry.py`, `_service/http_bridge/request_submit.py`, `_service/websocket/mixin.py`; normative spec `openspec/changes/archive/2026-07-12-add-security-work-account-routing/`.

### TA6(a) — Proactive cyber-binding (improvement over codex-lb's reactive-only) → **bind at CREDENTIAL/PROFILE scope, not session scope; make the session requirement sticky**
codex-lb is purely reactive (waste one upstream round-trip, then retry). PolyFlare adds proactive binding so cyber-scoped work hits a trusted account from turn 1. A request's `required_capabilities` resolves by precedence:
1. **Request-level** — a header (`X-PolyFlare-Capability: security_work`) or a model alias (`…-cyber`). The Codex/Claude CLI can carry this via its provider `http_headers` config, so a dedicated "cyber" CLI profile tags every request. Most specific.
2. **Credential / client-profile-level** — dashboard-managed: "API key `pf_cyber_…` / named client profile always requires `security_work`." **This is the answer to the session-naming problem** (below): known on request 1, human-nameable → true from-start binding.
3. **Session sticky-flag** — once a session is determined to need cyber (proactively OR after a reactive rejection), persist `required_capabilities` on the session/continuity state so ALL later turns pre-filter to trusted accounts. Concrete improvement over codex-lb, which re-reacts every request.
4. **Reactive fallback** — the TA6 upstream-rejection path, unchanged, as the safety net.
- **Why credential-scope for from-start, not session-scope:** at the proxy, PolyFlare only sees an opaque `session_id` (derived from headers / content-hash), never a human name — the operator can't pick "the session about X" before it exists, and by the time a session appears in the dashboard its first request already left. API keys / client profiles ARE human-nameable and present on request 1 → the correct hook for "this whole stream of work is cyber."
- **Dashboard session-grain pin (secondary tool):** to pin an already-running conversation you forgot to pre-declare, the dashboard identifies live sessions by **content-preview + model + recency + current account** (better-ccflare's conversation viewer), not a name. Works turn 2+, not turn 1.
- **End-to-end:** cyber CLI profile (header / cyber API key) → PolyFlare resolves `required_capabilities={security_work}` → Selector pre-filters to trusted pool from turn 1 → session sticky-flag keeps it there. Same shape for Claude via the Anthropic executor.
- **Scope:** M2/M3 (capability resolution + selector eligibility + session sticky-flag + per-API-key capability config); dashboard UI is L5; Anthropic specifics M4.
- **Revisit if:** clients can't set headers/aliases in practice → lean on credential-scope; or a session-label header becomes standard → could name sessions directly.

### TA6(b) — pinned requests DO move on a hard capability rejection → **REVERSES TA6 point 5** (user-directed, 2026-07-17; security-sensitive, explicitly signed off)
**Field evidence:** in codex-lb, a session that hit the cyber requirement while sticky/pinned got **stuck** — it never moved to a cyber-capable account, it just kept failing on the pinned owner. That is exactly what TA6 point 5 ("PINNED requests are NEVER moved → surface the error unchanged") prescribes. The incident is direct evidence that rule is wrong.
- **Why TA6.5's premise is now obsolete:** it rested on *"moving a pinned request would re-wedge"* — i.e. the belief that dropping the anchor and re-sending is expensive/dangerous. The live transport measurements ([[handover-prefill-experiment]], `TRANSPORT-FINDINGS-2026-07-17.md`) **disproved** that: an HTTP-SSE full resend to another account is cheap and wedge-immune (~free, only an org-cache miss), and on WS it is a fast catchable error, not a hang. So "never move" now trades a **cheap resend** for a **guaranteed client-facing failure** (the owner rejects *every* cyber turn) — a bad trade. The wedge that justified the rule only ever lived on a narrow WS-anchor path, not here.
- **The revision:** a pinned/continuity-owned session that hits a **hard capability rejection** (cyber) its owner can't satisfy **DOES move** to a capability-holding account: strip the (worthless-for-this-turn) anchor → re-select from accounts holding the cap → full resend → `record_recovery` re-homes session ownership to the new account → stamp the **session sticky-flag** (TA6(a).3) so every later turn pre-filters to capable accounts (pay the reject-and-move cost ONCE). Capability-holding accounts are a **superset** (they also serve non-cyber), so staying there after the move costs nothing.
- **Not a security bypass — it falls the RIGHT way:** moving *to a cyber-authorized account* routes to the account that IS authorized; the invariant "never serve cyber work on a non-authorized account" is preserved. This is the opposite of falling open.
- **Security floor — REVISES codex-lb degrade step 4 for the cyber case (user chose STRICT):** when a session is sticky-cyber and NO capable account is available (all rate-limited/down), **fail with a clear "no authorized account available" error — NEVER fall back to unfiltered failover onto a non-authorized account.** codex-lb's documented degrade step 4 (warn + unfiltered failover) is correct for a *capacity* miss but wrong for a *capability* miss: on a cyber-flagged request an unauthorized account only rejects anyway, so falling back buys a muddier failure, not success. Cyber work waits or fails; it never lands on a non-authorized account.
- **Dedicated "cyber pool" (proactive, user-requested):** tag a **pool** with a pool-level `required_capability = security_work`. Any request routed there (`/cyber/responses` or a `pf_cyber_…` key) auto-resolves `required_capabilities={security_work}` from turn 1, and the selector pre-filters to the pool's capability-holding accounts. **Self-enforcing:** the pool tag declares the requirement, the per-account flag declares who satisfies it — a non-cyber account accidentally placed in the pool is still excluded by the capability filter. This is TA6(a)'s proactive precedence-1/2 hook expressed as a pool. So both paths exist: tag-a-pool-cyber for known-cyber work (zero wasted round-trips), and reactive auto-move+sticky for sessions that only reveal the need mid-stream.
- **Dependency + sequencing:** detecting the rejection needs the upstream **error-code string** in the failure signal — the SAME enabler A6 (quota) and A7 (auth-writeback) need (`FailureSignal`/`ExecError::UpstreamStatus` carries only `{status, retry_after}` today; see PORTING Phase A audit). So cyber-move rides the exact A6/A7 work and is **gated behind M5a** (which is currently reshaping that `ExecError` path — concurrent edits would collide). One enabler unlocks three routing behaviors.
- **Revisit if:** a capability rejection ever proves NOT reliably classifiable from the error code/body → fall back to the reactive-only detection TA6 already documents (unchanged as the safety net).

## M2-GATE1 — trait seam shapes are PROVISIONAL, not frozen (reshape at M2 kickoff) — from M1 whole-branch review
The M1 spec called the five trait signatures "fixed seams." The M1 whole-branch review (opus) flags that three are the wrong shape and will force changes once M2/M3 build real logic on them. **Decision: drop the "frozen" label for these three; the FIRST M2 design gate is reshaping them with real requirements in hand.** `Translator` and `Executor` are well-shaped (async+Result / pure transform) and hold.
- **`Selector::pick(&self, &[Account], &RequestCtx) -> Option<&Account>`** — inputs too thin. Real selection scores on live per-account runtime state (in-flight count, rate-limit remaining, cooldown, health) + continuity ownership + TA6 capability eligibility — none reachable from static `Account` (id/base_url/token) + `RequestCtx{session_id}`. `&[Account]` also forces a contiguous Vec pool (rules out a map-keyed pool). M2 must enrich the inputs before code depends on it.
- **`Continuity::prepare(&self, req, ctx) -> PreparedRequest`** — sync, infallible, store-less. The M3 continuity state machine (anchor/trim/resend — the wedge-prone core, [[sol-anchor-wedge-rootcause]]) MUST read/write persisted session state (async sqlx), can fail, and per TA6(a) stamps a session sticky-flag. Almost certainly becomes `async fn prepare(...) -> Result<PreparedRequest, ContinuityError>` with a session-state handle. Highest-confidence change.
- **`Coordinator::admit(&self, ctx) -> bool`** — admission + session-ownership typically must `await` (acquire permit / claim ownership) and return an RAII guard whose `Drop` releases the slot. `-> bool` expresses neither; likely `async fn admit(...) -> Result<AdmissionGuard, Rejected>`.
- **Why provisional, not reshaped now:** nothing implements these in M1, and shaping async/Result/guard signatures without M2's actual requirements is speculation (YAGNI). Reshape at M2 design — cheap (in-workspace, zero dependents today).
- **M4 note:** `Translator::translate_response_event(&self, Value) -> Value` is stateless 1-in-1-out; a real non-identity cross-format streaming translator (Anthropic↔OpenAI) often needs `&mut self` state + 1→N output. Revisit when the seam meets its first non-identity translator.

## M2-SCOPE — split M2 into M2a (persistence) + M2b (routing brain)
M2 (store + accounts + selector + OAuth import) is large; split into two shippable, independently-mergeable halves:
- **M2a** — SQLite store (`sqlx`) + schema (accounts, usage_history) + XChaCha20-Poly1305 token crypto + account repository + zero-re-auth OAuth **importer** (Fernet-decrypt the codex-lb blobs → re-encrypt XChaCha). Ships: `polyflare accounts import` works, accounts persist + load.
- **M2b** — OAuth **refresh** (decode-only JWT claims + `POST /oauth/token` + should_refresh 8-day) + the ported default **`capacity_weighted` selector** (pure scoring core + parity tests) + minimal snapshot assembly + **pool wiring** into ingress (replace M1's single hardcoded account). Ships: multi-account routing.
- Deferred to later/data-driven: live health-tier + in-flight runtime tracking, the other 7 selector strategies. Port reference: `docs/reference/codex-lb-port-reference.md`.

### M2-GATE1 (refined) — reshape ONLY `Selector` in M2; `Continuity`/`Coordinator` at their own milestones
YAGNI: reshape a seam when its milestone actually builds on it. M2b implements `Selector`, so reshape it NOW to `fn pick(&self, candidates: &[AccountSnapshot], ctx: &SelectionCtx) -> Option<AccountId>` — rich per-account snapshot input (status/used%/secondary%/reset_at/capacity/routing_policy/health_tier/error_count/cooldown/last_selected/in-flight/plan/security_work_authorized), **owned `AccountId` return**, and it stays **sync** (scoring is pure given a snapshot; the async DB snapshot-assembly lives in the caller/pool, not the trait — so `Selector` does NOT need to be async, only richer-input + owned-return, which is exactly the reviewer's core point). `Continuity` (M3) and `Coordinator` (their milestone) stay as-is but PROVISIONAL — reshape to `async fn → Result` with real requirements then; do not invent `ContinuityError`/`AdmissionGuard` speculatively now. Supersedes M1's blanket "reshape all three at M2 kickoff": only `Selector` has a concrete M2 consumer.
